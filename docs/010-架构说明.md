# my-trend 架构说明
**创建时间**: 20260320 00:00

## 系统定位

A股多维度舆情与分析系统。采集市场热度、新闻、股吧情绪数据，结合 LLM 大模型对个股、行业、板块进行结构化评分分析，输出用于选股信号和投资决策的数据。

## 包结构与职责

系统由 6 个独立包组成，每个包三层结构：

```
{包名}/
├── analyzer.py / fetcher.py   能力层：纯数据采集 / API调用 / LLM调用，不关心调度
├── models.py                  数据层：SQLAlchemy 表模型定义
└── main.py                    调度层：间隔控制、并发管理、断点续跑、入库编排
```

### 包一览

| 包 | 功能 | 数据源 | 分析对象 | 输出 |
|----|------|--------|---------|------|
| `heat` | 热度采集+分析 | 东方财富API + AkShare | 全市场~5500只 | 人气排名/关键词/历史趋势/热度Top20/分钟K线 |
| `news` | 新闻采集 | 今日头条 | 自选股 | 个股新闻文章 |
| `guba` | 股吧情绪 | 股吧帖子 + LLM情绪分类 | 单股/市场/全量 | 情绪评分(百分制) |
| `analysis` | 个股分析 | 今日头条 → LLM | 国际/国内/个股 | 多维评分(百分制) |
| `industry` | 行业分析 | 今日头条 → LLM | 申万L1 31个行业 | 行业多维评分(百分制) |
| `sector` | 板块分析 | 今日头条 → LLM | 东财dc_index ~1010个板块 | 板块多维评分(百分制) |

## 数据流

### 采集类（heat / news）

```
外部API → fetcher.py 解析 → main.py 限流调度 → batch_upsert → MySQL
```

### 分析类（analysis / industry / sector / guba）

```
今日头条搜索 → 新闻列表 → 构造prompt → 豆包LLM → JSON解析 → batch_upsert → MySQL
     ↑                                                              ↑
 串行搜索（限流5-10s）                                        并发LLM（3线程）
```

分析类统一模式：**串行搜索 + 并发 LLM**
- 主线程按顺序搜索新闻（每次间隔5s，防限流）
- 搜到后提交线程池做 LLM 分析（Semaphore 控制并发上限）
- 滑动窗口收割：主线程搜索下一个时同步回收已完成的 LLM future

## LLM 分析架构

### 统一评分体系

所有分析模块输出统一格式：

```json
{
  "sentiment": "乐观|中性|悲观",
  "summary": "综合评价（100-200字）",
  "score": 50,
  "scores": {
    "维度1": {"score": 50, "reason": "理由"},
    "维度2": {"score": 50, "reason": "理由"}
  },
  "key_events": [{"event": "...", "sentiment": "...", "date": "..."}]
}
```

- score 统一百分制，50分中性
- 每个分析对象每天一条记录，幂等写入

### 各模块评分维度

| 模块 | 维度（权重） |
|------|------------|
| 个股 analysis | growth(25%) + profitability(25%) + competitive(20%) + risk_level(15%) + catalyst(15%) |
| 行业 industry | policy(25%) + supply_demand(25%) + profitability(20%) + risk_level(15%) + catalyst(15%) |
| 板块 sector | policy(25%) + market_heat(25%) + profitability(20%) + risk_level(15%) + catalyst(15%) |

### 搜索策略

每个分析对象搜索2次今日头条：

| 模块 | 关键词1 | 关键词2 |
|------|---------|---------|
| 个股 | `{公司名}` | `{公司名} 项目 利润 行业前景 产品` |
| 行业 | `{行业名}行业` | `{行业名} 政策 产能 龙头 前景` |
| 板块 | `{板块名}板块` | `{板块名} 政策 资金 龙头 前景` |

## 与 my-stock 的协作

### 数据依赖

industry 和 sector 模块依赖 my-stock 数据库中的行情数据：

| my-trend 模块 | 依赖 my-stock 表 | 用途 |
|--------------|-----------------|------|
| industry | index_classify | 申万行业列表（L1 31个） |
| industry | sw_daily | 申万行业日行情（排名用） |
| sector | dc_index | 东财板块列表+日行情 |

通过 `my-stock/la/scripts/sync_remote.py` 每日同步：

```

                                ↑

```

## 反爬与容错

### 代理切换 (clash_proxy.py)

- 通过 Clash REST API 管理 36 个代理节点
- 搜索失败自动轮换 IP + 重建 HTTP session
- 测速筛选可用节点（<500ms）
- 所有节点耗尽时抛异常，任务中断但保留断点

### 断点续跑 (analysis/guba)

- `AnalysisRun` 表记录运行状态（run_id + 游标 + 失败数）
- 按 stock_code 排序固定顺序，游标记录已完成位置
- 中断后重跑自动从游标位置继续
- `--retry` 可重跑失败的股票

### 幂等写入

- `batch_upsert()`：ON DUPLICATE KEY UPDATE，重复更新
- `batch_insert_ignore()`：INSERT IGNORE，重复跳过
- 分批提交（500条/批），崩溃不丢已提交数据
